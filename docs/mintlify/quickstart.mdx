# Quickstart Guide

Get started with Ibex DB in minutes - either locally with Docker or using the production API.

## Production API

<Tip>
**Already deployed and ready to use!**

The production API is live with a clean, professional URL and API key authentication.
</Tip>

### Step 1: Get Your API Key

Contact your administrator to get an API key. You'll receive:
- **Production URL**: Your custom domain endpoint
- **API Key**: A secure key for authentication

### Step 2: Make Your First Request

```bash
curl -X POST "https://your-domain.com/path" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "operation": "LIST_TABLES",
    "tenant_id": "your-tenant",
    "namespace": "default"
  }'
```

### Step 3: Import Postman Collection

We provide complete Postman collections with 40+ pre-configured requests:

1. Import environment: `postman/environments/Production_CustomDomain.postman_environment.json`
2. Import collection: `postman/collections/Ibex_DB_AWS.postman_collection.json`
3. Set your API key in the environment
4. Start testing!

## Local Development

Run Ibex DB locally using Docker for development and testing.

### Prerequisites

- Docker & Docker Compose
- curl or Postman (for testing)

### Start the Service

<Steps>
  <Step title="Navigate to docker directory">
    ```bash
    cd docker
    ```
  </Step>

  <Step title="Start FastAPI service">
    ```bash
    docker compose up -d fastapi
    ```

    This starts:
    - FastAPI server (port 9000)
    - MinIO (S3-compatible storage)
    - Iceberg REST Catalog
  </Step>

  <Step title="Verify health">
    ```bash
    curl http://localhost:9000/health
    ```

    Expected response:
    ```json
    {
      "status": "healthy",
      "service": "Ibex DB",
      "version": "1.0.0"
    }
    ```
  </Step>
</Steps>

### Your First Local Request

<Steps>
  <Step title="Create a Table">
    ```bash
    curl -X POST http://localhost:9000/database \
      -H "Content-Type: application/json" \
      -d '{
        "operation": "CREATE_TABLE",
        "tenant_id": "my-tenant",
        "namespace": "default",
        "table": "users",
        "schema": {
          "fields": {
            "id": {"type": "long", "required": true},
            "name": {"type": "string", "required": true},
            "email": {"type": "string", "required": false},
            "age": {"type": "integer", "required": false}
          }
        }
      }'
    ```
  </Step>

  <Step title="Insert Records">
    ```bash
    curl -X POST http://localhost:9000/database \
      -H "Content-Type: application/json" \
      -d '{
        "operation": "WRITE",
        "tenant_id": "my-tenant",
        "table": "users",
        "records": [
          {"id": 1, "name": "Alice", "email": "alice@example.com", "age": 30},
          {"id": 2, "name": "Bob", "email": "bob@example.com", "age": 25}
        ]
      }'
    ```
  </Step>

  <Step title="Query Data">
    ```bash
    curl -X POST http://localhost:9000/database \
      -H "Content-Type: application/json" \
      -d '{
        "operation": "QUERY",
        "tenant_id": "my-tenant",
        "table": "users",
        "filter": {"age": {"gte": 25}},
        "sort": [{"field": "age", "order": "DESC"}]
      }'
    ```

    **Response:**
    ```json
    {
      "success": true,
      "data": [
        {
          "_tenant_id": "my-tenant",
          "_record_id": "abc123",
          "_timestamp": "2025-11-14T21:00:00Z",
          "_version": 1,
          "_deleted": false,
          "_deleted_at": null,
          "id": 1,
          "name": "Alice",
          "email": "alice@example.com",
          "age": 30
        }
      ],
      "metadata": {
        "row_count": 2,
        "execution_time_ms": 95.5,
        "scanned_bytes": 1024,
        "scanned_rows": 2,
        "cache_hit": false
      }
    }
    ```
  </Step>
</Steps>

## Understanding System Fields

Every record automatically includes audit fields:

| Field | Description |
|-------|-------------|
| `_tenant_id` | Tenant identifier for multi-tenancy |
| `_record_id` | Unique record identifier (MD5 hash) |
| `_timestamp` | Last modification timestamp |
| `_version` | Version number (increments on update) |
| `_deleted` | Soft delete flag |
| `_deleted_at` | Deletion timestamp |

These fields enable:
- **Version History**: Track all changes to a record
- **Time Travel**: Query data as it existed at any point in time
- **Audit Trail**: Complete compliance and audit capabilities
- **Soft Deletes**: Recover accidentally deleted data
